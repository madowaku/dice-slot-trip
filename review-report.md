Google Playリリースに向けた『Dice Slot Trip』レビューレポート

現在制作中のすごろくゲーム『Dice Slot Trip』のコードベースおよびプロジェクト設定を調査し、Google Playへのリリースにあたって重要となるポイントや改善項目をレビューしました。



全体的に、自動セーブや中断再開（ロールトランザクションのステート復旧）といったモバイルに不可欠なシステムが非常に堅牢に設計されており、単体テスト（150件以上）もすべて合格するなど、極めて高い品質で開発が進められています。



この品質を維持したまま、Google Playへのリリース手続きをスムーズに進め、ユーザーに快適に遊んでもらうためのチェックリストと具体的な提案を以下にまとめました。



1\. Androidエクスポート設定 (export\_presets.cfg) の見直し

Google Play Consoleへのリリースにあたり、現在開発用となっている設定を本番用に最適化する必要があります。



1.1 パッケージ名 (Unique Name) の本番化

現状: package/unique\_name="com.madowaku.diceslottrip.dev"

対策: リリース用に開発者用サフィックス (.dev) を除外した本番パッケージ名（例: com.madowaku.diceslottrip）に変更します。

WARNING



Google Play Consoleに一度アプリを登録すると、パッケージ名は二度と変更できません。慎重に決定してください。



1.2 リリース署名 (Release Keystore) の設定

現状: リリース署名のキーストアパスおよび認証情報が設定されていません。

対策: Javaの keytool または Android Studio を用いて専用のキーストア（.keystore または .jks）を作成し、エクスポート設定の Release 欄にパスとパスワードを設定する必要があります。

CAUTION



リリース用のキーストアファイルとパスワードは絶対に紛失しないように安全に保管してください。紛失すると、Google Play上のアプリのアップデートが一切できなくなります。



1.3 アプリアイコンと起動画面 (Splash Screen)

現状: launcher\_icons や splash\_screen のアセット指定が空欄になっています。

対策:

本番用のアプリアイコン（192x192、またはアダプティブアイコン用の前景/背景アセット）を設定します。

デフォルトのGodotロゴが表示されないよう、オリジナルの起動画面用画像を splash\_screen/icon または branding\_image に設定してください。

1.4 振動機能のパーミッション設定

現状: permissions/vibrate=false

対策: 仕様書や設計ノートには「弱い端末振動を入れられる」「振動オン・オフの設定」といった計画が見られます。将来的にゲーム内で振動機能（Input.vibrate\_handheld()）を実装・有効化する場合は、このパーミッションを true に設定する必要があります。

2\. アプリサイズ (AAB/APK) の最適化

モバイル環境ではダウンロードサイズがコンバージョン率に直結するため、不要なアセットの梱包を避ける必要があります。



2.1 不要な検証用画像の除外

懸念点: プロジェクトのルートディレクトリに、開発・確認用と思われる巨大な画像アセットが配置されています。

ステージ画面イメージ.png

ステージ画面イメージ3.png

ステージ選択画面イメージ.png

&#x20;(\~2.9MB)

タイトル画面イメージ.png

&#x20;(\~2.7MB)

影響: エクスポート設定が export\_filter="all\_resources"（すべてのリソースを同梱）になっているため、これらの合計約14MBの検証用画像がそのままリリース用パッケージに含まれてしまいます。

対策:

これらのファイルをプロジェクトディレクトリ外に退避させる。

または export\_presets.cfg の exclude\_filter に \*.png (ルート直下のみ) や特定のファイル名を追加して、エクスポート対象から除外する。

リソースのエクスポートフィルタを export\_selected\_resources に変更し、必要なシーンやアセットのみを明示的に選択してビルドする。

3\. メモリリーク警告の解消

単体テストの終了時に、Godotのメモリリーク警告が検出されています。



text



WARNING: 5 ObjectDB instances were leaked at exit (run with `--verbose` for details).

ERROR: 1 resources still in use at exit (run with --verbose for details).

懸念点: テストスクリプトの実行終了時やシーン切り替えの際に、動的生成された Object や Resource が正しく解放されていない可能性があります。

影響: Androidなどのモバイル環境ではメモリ管理が厳しく、リークが徐々に蓄積していくと、長時間のプレイにおいて OOM (Out Of Memory) による強制終了（クラッシュ）の原因になります。

対策:

動的に new() されたリファレンスが循環参照になっていないか確認し、ノードの場合は確実に queue\_free() されているか確認します。

テスト終了時のクリーンアップ処理（tests/run\_tests.gd など）で、テスト用にインスタンス化したノードやシングルトンが正しく破棄されているか再調査することを推奨します。

4\. コード構成とデバッグコードの分離

4.1 main.gd の肥大化とメンテナンス性

main.gd

&#x20;は250KB（4,400行以上）を超える巨大なスクリプトになっており、UI制御、状態遷移、QA用のデバッグ関数（\_qa\_...）がすべて混在しています。

リスク:

本番リリース後にバグが発生した際、影響範囲の特定や修正が難しくなります。

QA/デバッグ用のコード（環境変数を読み込んで動作を変えるロジックなど）が本番ビルドで意図せず発火したり、クラッシュを引き起こしたりするリスクがあります。

対策:

状態遷移（PlayScreenへの遷移など）やQA用テストランナー、デバッグUIを別々のスクリプト・シーンに分割し、main.gd の責務を軽量化する（リファクタリング）。

リリースビルド時には、デバッグ用のコードブロックやQA用関数が完全にバイパスされるよう、より厳密に OS.is\_debug\_build() でガードされているか再確認する。

5\. Google Playの必要要件クリア状況

5.1 Android SDK / Target API の確認

Google Playへのリリースには Target SDK 34 (Android 14) 以上 が必須です。

Godot 4.7 stable は標準で Target SDK 34 に対応していますが、もし古いカスタムエクスポートテンプレートや古いバージョンのSDKビルドツールを使用している場合は、ビルドエラーやストアでのアップロード拒否が発生します。

対策として、ビルド前にAndroidエクスポート設定でSDKターゲットが 34 以上になっていることを確認してください。

5.2 64ビット対応

エクスポート設定で architectures/arm64-v8a=true になっているため、64ビット要件はクリアしています。

古い端末や一部のエミュレータ（32ビット）をサポートする場合は armeabi-v7a もオンにできますが、現在のGoogle Playでは arm64-v8a のみが必須要件です。

推奨する次のステップ

不要アセットの整理: ルートにある ステージ画面2.png などのデバッグ用アセットをプロジェクト外へ移動する。

本番用エクスポート設定の作成: パッケージ名、キーストア、アプリアイコンを指定する。

実機デバッグビルドでの検証: AAB/APKをビルドし、Android実機に転送して、画面のセーフエリア（切り欠きやパンチホール）でUIが崩れていないか確認する。

リーク解析: Godotエディタのデバッガで「モニター」タブを開き、メモリやオブジェクト数が時間経過で無駄に増え続けていないか確認する。
